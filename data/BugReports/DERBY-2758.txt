               <div id="descriptionmodule_heading" class="mod-header">

                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>In Derby 10.2 and earlier an ODBC application which called the SQLForeignKeys function would return a set of "imported" and/or "exported" keys depending on the arguments passed in. For more on that function and its arguments, see:</p> 
                  <p> <a href="http://msdn.microsoft.com/library/default.asp?url=/library/en-us/odbc/htm/odbcsqlforeignkeys.asp" class="external-link" rel="nofollow">http://msdn.microsoft.com/library/default.asp?url=/library/en-us/odbc/htm/odbcsqlforeignkeys.asp</a></p> 
                  <p>In particular we have the following (pasted from the above link):</p> 
                  <p>&lt;begin paste&gt;</p> 
                  <p>If *PKTableName contains a table name, SQLForeignKeys returns a result set containing the primary key of the specified table and all of the foreign keys that refer to it. The list of foreign keys in other tables does not include foreign keys that point to unique constraints in the specified table.</p> 
                  <p>If *FKTableName contains a table name, SQLForeignKeys returns a result set containing all of the foreign keys in the specified table that point to primary keys in others tables, and the primary keys in the other tables to which they refer. The list of foreign keys in the specified table does not contain foreign keys that refer to unique constraints in other tables.</p> 
                  <p>If both *PKTableName and *FKTableName contain table names, SQLForeignKeys returns the foreign keys in the table specified in *FKTableName that refer to the primary key of the table specified in *PKTableName. This should be one key at most.</p> 
                  <p>&lt;end paste&gt;</p> 
                  <p>Note that either PKTableName or FKTableName could be missing, i.e. could be null.</p> 
                  <p>Now, in org/apache/derby/catalog/SystemProcedures.java, there is a static method called "SQLFOREIGNKEYS" which, to quote the javadoc, "map<span class="error">[s]</span> SQLForeignKeys to EmbedDatabaseMetaData.getImportedKeys, getExportedKeys, and getCrossReference".</p> 
                  <p>That method looks at some "options" that it receives from the client and makes a call to the corresponding method on EmbedDatabaseMetaData:</p> 
                  <p> String exportedKeyProp = getOption("EXPORTEDKEY", options);<br> String importedKeyProp = getOption("IMPORTEDKEY", options);</p> 
                  <p> if (importedKeyProp != null &amp;&amp; importedKeyProp.trim().equals("1"))<br> rs<span class="error">[0]</span> = getDMD().getImportedKeys(fkCatalogName,<br> fkSchemaName,fkTableName);<br> else if (exportedKeyProp != null &amp;&amp; exportedKeyProp.trim().equals("1"))<br> rs<span class="error">[0]</span> = getDMD().getExportedKeys(pkCatalogName,<br> pkSchemaName,pkTableName);<br> else<br> rs<span class="error">[0]</span> = getDMD().getCrossReference (pkCatalogName,<br> pkSchemaName,<br> pkTableName,<br> fkCatalogName,<br> fkSchemaName,<br> fkTableName);</p> 
                  <p>That said, when running with the DB2 Runtime Client the "options" argument only contains "ODBC"; it does not (appear to) contain "IMPORTEDKEY" nor "EXPORTEDKEY". So with that client we ultimately end up calling "getCrossReference()" every time. And in EmbedDatabaseMetaData.getCrossReference(), we see:</p> 
                  <p> PreparedStatement s = getPreparedQuery("getCrossReference");<br> s.setString(1, swapNull(primaryCatalog));<br> s.setString(2, swapNull(primarySchema));<br> s.setString(3, swapNull(primaryTable));<br> s.setString(4, swapNull(foreignCatalog));<br> s.setString(5, swapNull(foreignSchema));<br> s.setString(6, swapNull(foreignTable));<br> return s.executeQuery();</p> 
                  <p>That is to say, if either primaryTable or foreignTable is null, we swap it with a "%" to be used for pattern matching. Prior to the 10.3, that worked fine. With <a href="https://issues.apache.org/jira/browse/DERBY-2610" title="Queries in metadata.properties allow tablepattern for JDBC methods that do not allow patterns" class="issue-link" data-issue-key="DERBY-2610"><del>DERBY-2610</del></a>, though, the getCrossReference query in metadata.properties was changed to disallow pattern matching for the primary key:</p> 
                  <p>@@ -532,11 +532,15 @@<br> #<br> #param1 = pattern for the PRIMARY CATALOG name <br> #param2 = pattern for the PRIMARY SCHEMA name <br> -#param3 = pattern for the PRIMARY TABLE name <br> +#param3 = PRIMARY TABLE name <br> #<br> #param4 = pattern for the FOREIGN CATALOG name ('%' for getExportedKeys())<br> #param5 = pattern for the FOREIGN SCHEMA name ('%' for getExportedKeys())<br> #param6 = pattern for the FOREIGN TABLE name ('%' for getExportedKeys())<br> +# <a href="https://issues.apache.org/jira/browse/DERBY-2610" title="Queries in metadata.properties allow tablepattern for JDBC methods that do not allow patterns" class="issue-link" data-issue-key="DERBY-2610"><del>DERBY-2610</del></a>: did not change from pattern matching to "T2.TABLENAME=?" <br> +# because getExportedKeys uses this query with '%' for foreign table<br> +# Future: may want to add a new query for getExportedKeys to remove the<br> +# "T2.TABLENAME LIKE ?" pattern<br> getCrossReference=\<br> SELECT CAST ('' AS VARCHAR(128)) AS PKTABLE_CAT, \<br> PKTABLE_SCHEM, \<br> @@ -587,7 +591,7 @@<br> WHERE \<br> ((1=1) OR ? IS NOT NULL) \<br> AND S.SCHEMANAME LIKE ? \<br> AND T.TABLENAME LIKE ? \ &lt;-- removed w/ <a href="https://issues.apache.org/jira/browse/DERBY-2610" title="Queries in metadata.properties allow tablepattern for JDBC methods that do not allow patterns" class="issue-link" data-issue-key="DERBY-2610"><del>DERBY-2610</del></a><br> AND T.TABLENAME=? \ &lt;-- added with <a href="https://issues.apache.org/jira/browse/DERBY-2610" title="Queries in metadata.properties allow tablepattern for JDBC methods that do not allow patterns" class="issue-link" data-issue-key="DERBY-2610"><del>DERBY-2610</del></a><br> AND S.SCHEMAID = T.SCHEMAID \</p> 
                  <p>As a result, the ODBC "SQLForeignKeys" function now returns zero rows because there is no table whose name equals the "%" that we swapped in for the null.</p> 
                  <p>I'm not sure what the best fix for this should be. The JDBC API for getCrossReference() indicates that both primaryTable and foreignTable "must match the table name as it is stored in the database", so perhaps the bug is in SystemProcedures.java, where the SQLForeignKeys function is mapped to a getCrossReference() call that passes nulls. But one could argue that, given the lack of information in the "options" string received from the client (esp. the missing "IMPORTEDKEY" or "EXPORTEDKEY" keywords), SystemProcedures.java is actually doing the right thing. Either way, it's not immediately clear to me how this should best be resolved...</p> 
                  <p>I'm marking this as a 10.3 regression (with 10.3 fixin) since the behavior in the 10.3 trunk is different than what it was in previous releases. If anyone disagrees with this, please feel free to say so and/or update accordingly.</p> 
                  <p>Note: I don't think this behavior is technically covered by the release note for <a href="https://issues.apache.org/jira/browse/DERBY-2610" title="Queries in metadata.properties allow tablepattern for JDBC methods that do not allow patterns" class="issue-link" data-issue-key="DERBY-2610"><del>DERBY-2610</del></a> because a) the release note does not mention getCrossReference() (should it??), and b) the call to "getCrossReference()" is made internally; a user's ODBC app would be calling SQLForeignKeys, for which a null primary/foreign table name are in fact perfectly legal.</p>
