                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Derby violates the SQL Standard when an UPDATE statement SETs an identity column to the value DEFAULT. Derby does the right thing for INSERTs, however.</p> 
                  <p>For INSERTs, the 2011 Standard defines the meaning of DEFAULT in part 2, section 15.10 (Effect of inserting tables into base tables), general rule 2. For INSERTs, the DEFAULT value is the next value of the sequence generator which defines the identity column.</p> 
                  <p>For UPDATEs, the 2011 Standard defines the meaning of DEFAULT in part 2, section 15.13 (Effect of replacing rows in base tables), general rule 5. For UPDATEs, the DEFAULT value is also the next value of the sequence generator which defines the identity column.</p> 
                  <p>Note also that the 2011 Standard says that a GENERATED ALWAYS identity column can be SET to DEFAULT. Furthermore, that is the only explicit value accepted. See the 2011 Standard, part 2, section 14.15 (set clause list), syntax rule 5:</p> 
                  <p>{<br> "If &lt;set clause&gt; SC specifies an &lt;object column&gt; that references a column of which some underlying column is either a generated column or an identity column whose descriptor indicates that values are always generated, then the &lt;update source&gt; specified in SC shall consist of a &lt;default specification&gt;."<br> }</p> 
                  <p>What Derby actually does for UPDATEs is the following:</p> 
                  <p>1) If the column was declared GENERATED ALWAYS, then Derby raises a compile-time error saying that the value of an identity column can't be overridden.</p> 
                  <p>2) If the column was declared GENERATED BY DEFAULT, then Derby raises an execution time-error when trying to stuff a null into the column.</p> 
                  <p>Correcting this bug would result in backwardly incompatible behavior. However, I think that the incompatibility is minor: it would mean the successful run of statements which previously raised errors.</p> 
                  <p>I tripped across this problem while implementing the UPDATE action of the MERGE statement (<a href="https://issues.apache.org/jira/browse/DERBY-3155" title="Support for SQL:2003 MERGE statement" class="issue-link" data-issue-key="DERBY-3155"><del>DERBY-3155</del></a>). If we decide to fix this bug, we will want to make sure that the UPDATE actions of MERGE statements also correctly handle DEFAULT values for identity columns.</p> 
                  <p>The following script shows this problem:</p> 
                  <p>connect 'jdbc:derby:memory:db;create=true';</p> 
                  <p>create table t1( a int generated always as identity, b int );<br> create table t2( a int generated by default as identity, b int );</p> 
                  <p>insert into t1( a, b ) values ( default, 100 );<br> insert into t2( a, b ) values ( default, 100 );</p> 
                  <p>update t1 set a = default;<br> update t2 set a = default;</p> 
                  <p>select * from t1;<br> select * from t2;</p> 
                 </div> 
                </div> 
               </div>
