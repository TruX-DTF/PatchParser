                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>PrefixCodedTermsReader/Writer (used by all "real" core codecs) already<br> supports pluggable terms index impls.</p> 
                  <p>The only impl we have now is FixedGapTermsIndexReader/Writer, which<br> picks every Nth (default 32) term and holds it in efficient packed<br> int/byte arrays in RAM. This is already an enormous improvement (RAM<br> reduction, init time) over 3.x.</p> 
                  <p>This patch adds another impl, VariableGapTermsIndexReader/Writer,<br> which lets you specify an arbitrary IndexTermSelector to pick which<br> terms are indexed, and then uses an FST to hold the indexed terms.<br> This is typically even more memory efficient than packed int/byte<br> arrays, though, it does not support ord() so it's not quite a fair<br> comparison.</p> 
                  <p>I had to relax the terms index plugin api for<br> PrefixCodedTermsReader/Writer to not assume that the terms index impl<br> supports ord.</p> 
                  <p>I also did some cleanup of the FST/FSTEnum APIs and impls, and broke<br> out separate seekCeil and seekFloor in FSTEnum. Eg we need seekFloor<br> when the FST is used as a terms index but seekCeil when it's holding<br> all terms in the index (ie which SimpleText uses FSTs for).</p> 
                 </div> 
                </div> 
               </div>
