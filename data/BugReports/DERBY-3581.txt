                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The client driver has code propagating changes made to the DataSource to existing connections created by that DataSource.<br> There is some discussion of this in the thread <a href="http://www.nabble.com/ConnectionPoolDataSource-properties-td15740457.html" class="external-link" rel="nofollow">http://www.nabble.com/ConnectionPoolDataSource-properties-td15740457.html</a>, and there is also an example of what can happen due to this "feature".</p> 
                  <p>Besides from being a bug with the potential to cause strange errors in deployment, the issue also makes the client driver code harder to read and understand.<br> As far as I can see, there is also some related code in other parts of the client driver, for instance for "fully" resetting statements. There is mention of dynamic versus static sections in the comments (this one from am.Statement):</p> 
                  <p> // If a dataSource is passed into resetClientConnection(), then we will assume<br> // properties on the dataSource may have changed, and we will need to go through<br> // the open-statement list on the connection and do a full reset on all statements,<br> // including preparedStatement's and callableStatement's. This is because property<br> // change may influence the section we allocate for the preparedStatement, and<br> // also the cursor attributes, i.e. setCursorSensitivity().<br> // If no dataSource is passed into resetClientConnection(), then we will do the<br> // minimum reset required for preparedStatement's and callableStatement's.</p> 
                  <p>Note that the reset code for statements is also invoked when ClientPooledConnection.getConnection() is invoked. I do not understand why we should reset statements when we get a new logical connection.</p> 
                  <p>Further, I also suspect the concept of a deferred reset has been introduced because of the feature/bug described by this Jira issue. A deferred reset seems to be a mechanism to avoid a round-trip to validate the newly changed DataSource properties (typically user, password and security mechanism).<br> I will look into removing code related to deferred resets as well. If you have historic information about these parts of the driver, I would appreciate if you share it with the community if possible.</p> 
                  <p>Just to be clear, it is my opinion that changed DataSource properties should take effect when one of the following methods is invoked:</p> 
                  <ul> 
                   <li>DataSource.getConnection()</li> 
                   <li>ConnectionPoolDataSource.getPooledConnection()</li> 
                   <li>XADataSource.getXAConnection()</li> 
                  </ul> 
                  <p>All of the methods above returns a physical connection. Properties like user name and password are associated with the physical connection, and thus requests to obtain a logical connection should not cause any of these properties to change.</p> 
                 </div> 
                </div>
