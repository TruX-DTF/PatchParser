                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>I see a problem in the code handling lock timeout exceptions. In the code in various places there are calls such as:</p> 
                  <p> // 2 kinds of errors here expected here. Either container not <br> // found or could not obtain lock (LOCK_TIMEOUT or DEADLOCK).<br> //<br> // It is possible by the time this post commit work gets scheduled <br> // that the container has been dropped and that the open container <br> // call will return null - in this case just return assuming no <br> // work to be done.</p> 
                  <p> if (se.getMessageId().equals(SQLState.LOCK_TIMEOUT) ||<br> se.getMessageId().equals(SQLState.DEADLOCK))</p> 
                  <p>Or </p> 
                  <p> // First try to do the work in the nested transaction. Fail if we can't<br> // get a lock immediately.<br> if ( nestedTransaction != null )<br> {<br> try </p> { return updateCurrentValueOnDisk( nestedTransaction, oldValue, newValue, false ); } 
                  <p> catch (StandardException se)<br> {<br> if ( !se.getMessageId().equals( SQLState.LOCK_TIMEOUT ) ) </p> { throw se; } 
                  <p> }</p> 
                  <p>Or</p> 
                  <p> // exception might have occured either container got dropper or lock not granted.<br> // It is possible by the time this post commit work gets scheduled <br> // that the container has been dropped and that the open container <br> // call will return null - in this case just return assuming no <br> // work to be done.</p> 
                  <p> //If this expcetion is because lock could not be obtained , work is requeued.<br> if (se.getMessageId().equals(SQLState.LOCK_TIMEOUT) || <br> se.getMessageId().equals(SQLState.DEADLOCK))</p> { requeue_work = true; } 
                  <p>The problem that I see is that if the property "derby.locks.deadlockTrace=true" is set, then instead of a SQLState.LOCK_TIMEOUT, the code will see a SQLState.LOCK_TIMEOUT_LOG. Note that this is not being checked for in the above tests and others as well, so now the code behavior is going to change basd on whether the lock tracing is enabled or not. </p> 
                  <p>I think that 99% of the code that is testing for SQLState.LOCK_TIMEOUT should also be checking for SQLState.LOCK_TIMEOUT_LOG as well. I only see one place in DDLConstantAction where it is explicitly mentioned that SQLState.LOCK_TIMEOUT_LOG is not being looked at.</p> 
                 </div> 
                </div>
