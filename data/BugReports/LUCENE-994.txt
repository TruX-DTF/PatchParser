                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>This is follow-through from <a href="https://issues.apache.org/jira/browse/LUCENE-845" title="If you &quot;flush by RAM usage&quot; then IndexWriter may over-merge" class="issue-link" data-issue-key="LUCENE-845"><del>LUCENE-845</del></a>, <a href="https://issues.apache.org/jira/browse/LUCENE-847" title="Factor merge policy out of IndexWriter" class="issue-link" data-issue-key="LUCENE-847"><del>LUCENE-847</del></a> and <a href="https://issues.apache.org/jira/browse/LUCENE-870" title="add concurrent merge policy" class="issue-link" data-issue-key="LUCENE-870"><del>LUCENE-870</del></a>;<br> I'll commit this once those three are committed.</p> 
                  <p>Out of the box performance of IndexWriter is maximized when flushing<br> by RAM instead of a fixed document count (the default today) because<br> documents can vary greatly in size.</p> 
                  <p>Likewise, merging performance should be faster when merging by net<br> segment size since, to minimize the net IO cost of merging segments<br> over time, you want to merge segments of equal byte size.</p> 
                  <p>Finally, ConcurrentMergeScheduler improves indexing speed<br> substantially (25% in a simple initial test in <a href="https://issues.apache.org/jira/browse/LUCENE-870" title="add concurrent merge policy" class="issue-link" data-issue-key="LUCENE-870"><del>LUCENE-870</del></a>) because it<br> runs the merges in the backround and doesn't block<br> add/update/deleteDocument calls. Most machines have concurrency<br> between CPU and IO and so it makes sense to default to this<br> MergeScheduler.</p> 
                  <p>Note that these changes will break users of ParallelReader because the<br> parallel indices will no longer have matching docIDs. Such users need<br> to switch IndexWriter back to flushing by doc count, and switch the<br> MergePolicy back to LogDocMergePolicy. It's likely also necessary to<br> switch the MergeScheduler back to SerialMergeScheduler to ensure<br> deterministic docID assignment.</p> 
                  <p>I think the combination of these three default changes, plus other<br> performance improvements for indexing (<a href="https://issues.apache.org/jira/browse/LUCENE-966" title="A faster JFlex-based replacement for StandardAnalyzer" class="issue-link" data-issue-key="LUCENE-966"><del>LUCENE-966</del></a>, <a href="https://issues.apache.org/jira/browse/LUCENE-843" title="improve how IndexWriter uses RAM to buffer added documents" class="issue-link" data-issue-key="LUCENE-843"><del>LUCENE-843</del></a>,<br> <a href="https://issues.apache.org/jira/browse/LUCENE-963" title="Add setters to Field to allow re-use of Field instances during indexing" class="issue-link" data-issue-key="LUCENE-963"><del>LUCENE-963</del></a>, <a href="https://issues.apache.org/jira/browse/LUCENE-969" title="Optimize the core tokenizers/analyzers &amp; deprecate Token.termText" class="issue-link" data-issue-key="LUCENE-969"><del>LUCENE-969</del></a>, <a href="https://issues.apache.org/jira/browse/LUCENE-871" title="ISOLatin1AccentFilter a bit slow" class="issue-link" data-issue-key="LUCENE-871"><del>LUCENE-871</del></a>, etc.) should make for some sizable<br> performance gains Lucene 2.3!</p> 
                 </div> 
                </div> 
               </div>
